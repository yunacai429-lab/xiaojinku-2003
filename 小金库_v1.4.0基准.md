## 1. 文档用途

本文档是小金库 v1.4.0 的静态功能与技术基准。后续版本更新时，应以本文档和对应版本代码为起点：保留未被明确要求修改的功能，完成新增或调整内容后同步更新版本号、README 和基准文档，并生成新的 `小金库_v1.x.0基准.md`，不要覆盖旧版本基准。

## 2. 产品概述

小金库是一款面向个人使用的响应式记账网页应用，可在电脑和手机浏览器中使用。产品采用复古桌面软件式界面，支持日常收支记录、类别预算、月付还款、账户余额、存款管理、统计分析、个人资料和界面皮肤切换。

应用提供两套皮肤：

- 复古1：蓝色 Windows XP 风格。
- 复古2：灰银色传统桌面软件风格。

桌面端以应用窗口形式显示，左侧为固定导航，右侧为可滚动内容区。手机端隐藏左侧导航，使用底部导航进入首页、记账、明细、还款、预算、账户和设置。

## 3. 现有功能

### 3.1 登录与数据同步

- 支持邮箱和密码注册、登录及退出登录。
- 登录页支持用户主动勾选“记住用户名和密码”；成功登录后将账号信息保存在当前浏览器，退出后再次进入登录页时自动填充。
- 未勾选时不保存登录凭据；取消勾选并成功登录后会清除此前保存的凭据。
- 使用 Supabase Authentication 管理账号。
- 登录后将完整账本保存到 Supabase，并支持不同设备间同步。
- 云端通过 Row Level Security 隔离不同用户的数据。
- 本地使用 `localStorage` 保存当前账号的账本缓存和用户主动选择保存的登录信息，使用 `sessionStorage` 保存会话辅助状态。
- 提供免登录演示模式，演示数据不会上传云端。
- 支持手动刷新并重新拉取云端数据。

### 3.2 首页与账户总览

- 显示本月收入、本月花销、本月剩余预算和待还款金额。
- 本月收入、本月花销和待还款卡片右侧提供按钮，可跳转到明细页并自动带入当前月份及对应类型筛选。
- 收入和花销显示与上个月相比的百分比变化；上个月无数据时显示 0.0%。
- 显示账户总余额、存款余额及各余额账户的当前余额。
- 类别预算、最近明细和存款流水展示全部数据，并在各自模块内部独立上下滚动。
- 收支明细按日期和时间倒序排列。
- 支持将指定月份的可用结余转入存款。
- 支持从存款提取到指定余额账户。
- 支持手动校准存款余额，并记录调整流水。
- 显示最近的存款转入、提取和调整流水。

### 3.3 记一笔

- 支持记录收入和花销。
- 支持填写金额、类别、日期、时间和备注。
- 类别、付款方式和支付平台旁提供绿色“增加+”按钮，可跳转到对应管理页面新增选项。
- 花销可选择余额支付或月付，并选择具体支付平台。
- 收入可选择具体入账账户。
- 收入、余额支付和已还月付会自动影响对应账户余额。

### 3.4 收支明细

- 支持按月份筛选。
- 支持按类型筛选：全部、支出、收入或待还款。
- 支持按消费类别筛选。
- 月份、类型和类别可组合选择，点击“筛选”后执行查询。
- 从首页指标卡进入时自动展示对应的当前月份筛选结果。
- 记录按日期和时间倒序排列，最新记录显示在最上方。
- 支持编辑和删除已有记录。
- 编辑或删除记录后同步校正相关账户余额。
- 页面外壳保持固定，明细列表独立上下滚动。

### 3.5 收支统计

- 支持按月、按年和自定义日期范围统计。
- 显示收入、花销、结余和记录数量。
- 使用柱状图比较收入与花销。
- 使用饼图和图例展示消费类别占比。

### 3.6 待还款

- 自动汇总使用月付方式产生且尚未还款的消费。
- 设置页可指定每月 1 至 31 号中的任意一天为还款日。
- 待还款页显示明确的“下次还款日为 x年x月x日，还有 x天”，不是日期范围。
- 当设定日期在某月不存在时，按该月最后一天计算。
- 到达本月还款日后，下次还款日自动计算为下个月对应日期。
- 按月付平台分类展示待还记录。
- 每个分类支持全选和取消全选。
- 支持选择多笔记录后，从指定余额账户一键还款。
- 还款后同步减少付款账户余额并更新记录状态。

### 3.7 类别与预算

- 支持新增、删除和重命名消费类别。
- 每个消费类别均可自定义 emoji 图标，不限于修改名称。
- 新增类别时同时填写类别名称、emoji 和预算。
- 重命名类别后同步更新历史记录中的类别名称。
- 删除类别时同步移除对应预算和图标配置，但保留历史交易记录。
- 支持拖动调整类别顺序。
- 支持为每个类别设置月度预算。
- 实时计算各类别已用金额和剩余预算。
- 手机版保留完整预算入口，类别、emoji、预算和操作按钮采用紧凑的单行编辑布局。

### 3.8 账户与支付方式

- 支持管理余额账户和月付方式。
- 支持新增、删除、重命名和拖动排序。
- 余额账户可直接校准当前余额。
- 支付方式重命名后同步更新历史记录、还款记录和存款流水。

### 3.9 设置与个人资料

- 设置页集中管理个人资料入口、每月还款日和界面皮肤。
- 支持将每月还款日设置为 1 至 31 的整数并同步到待还款页面。
- 支持在复古1和复古2之间即时切换，选择结果自动保存。
- 登录用户的皮肤和还款日设置随账本同步到云端。
- 支持修改用户名。
- 支持从本地相册选择头像、裁切为方形并压缩保存。
- 支持恢复默认头像。

### 3.10 顶部工具栏

- “记一笔”进入记账页面。
- “报表”进入统计页面。
- “设置”进入设置页面。
- 新建、编辑、删除、导入、导出、备份和帮助当前为灰色不可用状态。
- 文件、编辑、查看和帮助菜单文字仅作界面展示，当前无下拉操作。

## 5. 技术栈

### 5.1 前端

- HTML5。
- CSS3：Grid、Flexbox、媒体查询和响应式固定布局。
- 原生 JavaScript，无前端框架、无打包器。
- 单页应用结构，由 `src/main.js` 根据内存状态生成 HTML 并绑定交互。
- 浏览器 `localStorage` 和 `sessionStorage`。
- Canvas API，用于头像裁切与压缩。
- Fetch API，用于访问 Supabase Authentication 和 REST API。

### 5.2 云端服务

- Supabase Authentication：邮箱密码注册和登录。
- Supabase PostgreSQL：保存用户账本数据。
- Supabase REST API：读取和更新账本。
- Row Level Security：限制用户只能访问自己的数据。
- Cloudflare Pages：托管正式静态网站。
- GitHub：保存源码和版本历史。

### 5.3 数据结构

每个用户在 Supabase 的 `ledger_data` 表中对应一行：

- `user_id`：Supabase 用户 ID，主键。
- `data`：完整账本 JSON。
- `updated_at`：最近更新时间。

账本 JSON 的主要字段：

- `profile`：用户名、头像、皮肤、每月还款日。
- `accounts`：余额账户及余额。
- `categories`：类别名称和显示顺序。
- `categoryIcons`：类别名称到自定义 emoji 的映射。
- `platforms`：余额账户与月付方式配置。
- `budgets`：各类别的月度预算。
- `transactions`：收入、花销、月付和还款状态明细。
- `savings`：存款余额。
- `savingsTransfers`：存款转入、提取和调整流水。

| 文件 | 用途 |

## 6. 已知限制

- 账本以单个 JSON 对象整体保存，数据量很大时同步和渲染性能会下降。
- 云端同步采用整份数据覆盖，多台设备同时编辑时可能发生最后一次保存覆盖较早修改。
- 没有服务端分页；明细、统计、预算和还款计算均在浏览器完成。
- 演示模式不写入云端，刷新或重新打开页面后需要重新进入演示模式。
- “记住用户名和密码”会把用户明确选择保存的凭据存入当前浏览器本地存储；共享设备不应勾选。
- 顶部的新建、编辑、删除、导入、导出、备份和帮助功能尚未开放。
- 文件、编辑、查看和帮助菜单目前没有下拉菜单或操作。
- 不支持多账本、家庭共享、多人协作和权限分级。
- 不支持账单附件、票据识别、自动银行同步和周期记账。
- 没有独立离线同步队列；网络恢复后可能需要重新保存或手动刷新。
- 头像依赖浏览器图片解码能力，部分浏览器可能无法直接读取 HEIC/HEIF。
- 当前不是可安装 PWA，没有离线安装包和系统级推送通知；还款提醒仅在应用页面内展示。
- 项目没有自动化测试和构建流程，发布前依赖手动功能及响应式检查。
- Supabase URL 和 publishable/anon key 配置在前端源码中；迁移 Supabase 项目时需手动替换。

## 7. 部署方式

### 7.1 本地预览

项目没有安装或构建依赖，在项目目录启动静态服务器：

```bash
cd xiaojinku-release
python3 -m http.server 4173
```

访问 `http://localhost:4173/`。不建议直接双击 `index.html`，以免浏览器限制本地资源请求。

### 7.2 Supabase 配置

1. 创建 Supabase 项目。
2. 在 SQL Editor 完整执行 `supabase-schema.sql`。
3. 在 `src/main.js` 顶部填写 `SUPABASE_URL` 和 `SUPABASE_KEY`。
4. `SUPABASE_KEY` 只能使用 publishable/anon key，不得使用 `service_role` key。
5. 在 Authentication 的 URL Configuration 中配置正式网站地址和允许的跳转地址。
6. 测试注册、登录、保存、刷新和跨设备同步。

### 7.3 Cloudflare Pages

正式网站部署在 Cloudflare Pages：`https://xiaojinku-2003.pages.dev/`。

项目为纯静态站点，可采用以下任一方式：

- Git 部署：连接 GitHub 仓库，构建命令留空，输出目录使用仓库根目录 `.`。
- Direct Upload：上传至少包含 `index.html` 和 `src/` 的静态目录。

当前仓库根目录就是可部署目录。发布后应确认 `index.html` 中的资源版本参数已更新，避免浏览器或 CDN 继续使用旧版缓存。

### 7.4 GitHub 与 Release

1. 更新 `VERSION`、`README.md`、`index.html` 的资源版本参数及新版基准文档。
2. 提交并推送到 GitHub 的 `main` 分支。
3. 使用对应版本号创建 Git tag，例如 `v1.4.0`。
4. 在 GitHub Releases 创建同版本 Release，设为 Latest，并填写本次更新说明。
5. 确认 Cloudflare Pages 已部署该提交，或执行 Direct Upload。
6. 打开正式网站核对资源版本和关键功能。

GitHub Pages、Netlify、Vercel 静态站点等也可托管，但必须支持 HTTPS，并保持 `/src/main.js` 和 `/src/style.css` 的路径不变。

## 8. 后续版本更新规则

- 新任务开始时先读取本基准文档、`README.md`、`VERSION` 和当前代码。
- 未在新需求中提及的功能和页面行为默认保持不变。
- 新功能升级次版本，例如 `v1.4.0` 到 `v1.5.0`；仅修复问题时升级修订版本，例如 `v1.4.0` 到 `v1.4.1`；不兼容的大改版才升级主版本。
- 同步更新 `VERSION`、README 当前版本、`index.html` 静态资源版本参数和发布说明。
- 为新版本生成新的基准文件，例如 `小金库_v1.5.0基准.md`，保留旧基准文档作为历史记录。
- 发布前检查电脑端和手机端，不得因桌面修改丢失手机导航或造成文字、按钮换行及重叠。
- 发布后同时核对 GitHub 源码、GitHub Release 和线上网站版本。
